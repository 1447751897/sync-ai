# AI 开发规则

本文件是项目开发时给 AI 助手读取并严格执行的统一规范。每次开始新项目或新任务前，AI 必须先阅读本文件，并在执行过程中持续遵守。

## 0. 使用方式

在每个新项目开始时，把本文件放到项目根目录，推荐命名为：

- `AI_DEVELOPMENT_RULES.md`
- `CLAUDE.md`
- `AGENTS.md`
- `.cursor/rules/project.md`

首次对话时使用以下提示：

```text
请先阅读项目根目录下的 AI_DEVELOPMENT_RULES.md，并在本次开发全过程中严格执行。除非我明确要求变更规则，否则不要跳过、弱化或自行改写这些约束。
```

## 1. 核心原则

1. 先理解，再修改。
2. 优先遵循现有项目结构、技术栈、命名风格和代码习惯。
3. 不做与当前任务无关的重构。
4. 不臆造不存在的接口、文件、字段、依赖或业务规则。
5. 遇到不确定的信息，先从代码、配置、文档、测试中查证。
6. 修改必须小步、可验证、可回退。
7. 完成后必须说明改了什么、为什么改、如何验证。

## 2. 开发前必须执行

开始写代码前，AI 必须完成以下检查：

1. 阅读项目结构，识别主要技术栈、入口文件、构建方式和测试方式。
2. 查找与当前需求相关的文件、组件、接口、状态管理、数据库模型或配置。
3. 如果已有相似功能，优先复用现有模式。
4. 如果需求不清楚，先提出最少数量的关键问题。
5. 如果可以根据上下文合理判断，则直接执行，不要频繁打断用户。

禁止：

- 不看代码直接给方案。
- 用全新架构覆盖已有架构。
- 为了实现小功能引入大型依赖。
- 修改无关文件。
- 删除用户已有改动。

## 3. 修改代码规则

AI 修改代码时必须遵守：

1. 保持改动范围最小。
2. 保持文件职责清晰，不把无关逻辑塞进同一个文件。
3. 优先使用项目已有工具函数、组件、样式变量、请求封装和错误处理方式。
4. 新增代码必须可读、可维护，避免炫技。
5. 不写无意义注释，只在复杂逻辑前添加简短说明。
6. 不硬编码敏感信息、密钥、账号、真实 token、生产地址。
7. 不吞掉错误，必须保留合理的错误处理和用户反馈。
8. 不为了让测试通过而降低业务正确性。

## 4. 前端开发规则

如果任务涉及前端，必须遵守：

1. 先读取 `docs/product/15-frontend-design.md` 和 `docs/product/15-frontend-design-tokens.json`；如果缺失或内容为空，必须先根据项目类型补齐 UI 风格参考、设计关键词和 token 值。
2. 所有颜色、间距、圆角、字体、阴影必须引用 `15-frontend-design-tokens.json` 中的 token 值，禁止硬编码随意值。如果需要新增 token，先添加到 token 文件再引用。
3. 优先匹配现有设计系统、组件库、间距、字体、颜色和交互习惯。
3. 页面必须在桌面端和移动端都不出现明显布局错乱。
4. 文本不能溢出按钮、卡片、表格单元格或弹窗。
5. 表单必须有明确的校验、错误提示和提交状态。
6. 异步数据必须处理 loading、empty、error、success 状态。
7. 交互控件要符合用户直觉：开关用 switch/checkbox，选项用 select/tabs/segmented control，危险操作要二次确认。
8. 不做空洞的营销页，除非需求明确要求。
9. 不添加与业务无关的装饰性元素。
10. 产品形态、UI 风格、布局、组件策略或交互状态发生变化时，必须反向更新 `docs/product/15-frontend-design.md`。

## 5. 后端开发规则

如果任务涉及后端，必须遵守：

1. API 入参必须校验。
2. 权限、身份、租户隔离等安全边界必须显式处理。
3. 数据库写入必须考虑事务、一致性和失败回滚。
4. 错误返回必须稳定、可诊断，不暴露敏感信息。
5. 日志必须有助于排查问题，但不能记录密码、token、身份证、银行卡等敏感信息。
6. 新增接口必须保持响应结构与项目既有风格一致。
7. 涉及分页、排序、搜索时，要明确默认值和边界条件。

## 6. 测试与验证规则

### 6.1 测试先行（TDD，项目有测试框架时默认启用）

如果项目已配置测试框架（jest/vitest/pytest/go test/junit 等），实现新功能或修复 bug 时必须遵循：

1. 先写失败测试（Red）：根据需求或 bug 编写最少验证用例，运行确认失败。
2. 最小实现（Green）：写刚好让测试通过的代码，不提前优化。
3. 重构（Refactor）：在测试保护下优化命名、抽象、性能，确保测试仍然通过。
4. 确认所有相关测试通过后，再进入下一步。

如果项目没有测试框架，跳过本步骤但在最终回复中标注"本项目未配置测试框架，建议后续补充"。

如果是纯 UI 调整、文档变更或配置修改等不适合单测的场景，跳过 TDD 但必须在验证门禁中提供替代证据。

### 6.2 验证门禁

实现完成后，AI 必须提供以下证据（至少满足 2 项）才能标记任务完成：

- [ ] 测试通过输出（单测、集成、E2E 任一）
- [ ] 构建成功输出（build/compile 无 error）
- [ ] 类型检查通过（tsc/mypy/go vet 等）
- [ ] 接口响应示例（curl/httpie/浏览器 devtools）
- [ ] 页面手动测试步骤（见 §9.1）

无法提供证据时，必须说明原因和剩余风险，不能默认标记完成。

### 6.3 基本验证流程

完成修改后，AI 必须尽可能执行验证：

1. 优先运行与改动相关的最小测试。
2. 如果有 lint/typecheck/build/test 命令，应根据改动风险选择运行。
3. 前端视觉改动需要启动页面并检查实际效果。
4. 修 bug 时必须说明复现原因和验证方式。
5. 如果无法运行测试，必须明确说明原因和剩余风险。

推荐验证顺序：

```text
相关单测 -> 类型检查 -> lint -> build -> 手动功能验证
```

## 7. Git 与文件安全规则

AI 必须遵守：

1. 不执行 `git reset --hard`。
2. 不执行会丢失用户改动的 checkout/revert 操作，除非用户明确要求。
3. 不删除不理解的文件。
4. 不改动锁文件，除非依赖确实发生变化。
5. 不提交代码，除非用户明确要求。
6. 遇到工作区已有改动时，默认认为是用户改动，必须保留。

## 8. 依赖与配置规则

1. 新增依赖前必须说明理由。
2. 能用项目已有依赖解决的问题，不新增依赖。
3. 不随意升级框架、构建工具、语言版本。
4. 配置改动必须说明影响范围。
5. 环境变量必须提供示例，不提交真实密钥。

## 8.0 技术选型确认规则

当任务涉及以下内容时，AI 不能直接决定技术方案，必须先给出方案推荐并等待用户确认：

1. 前端框架、后端框架、构建工具、运行时。
2. 数据库、缓存、消息队列、搜索引擎、对象存储。
3. 鉴权、权限、支付、邮件、短信、推送、第三方登录。
4. 部署平台、CI/CD、容器化、云服务、监控、日志、任务调度。
5. 图表、富文本、地图、表格、文件预览等重型前端依赖。
6. 会影响架构、性能、成本、安全、维护方式的新增依赖或服务。

技术选型流程必须是：

1. 说明当前技术需求和约束。
2. 给出 2-3 个可选方案。
3. 对比每个方案的目的、优点、缺点、性能影响、成本和维护复杂度。
4. 给出推荐方案和推荐理由。
5. 等待用户确认。
6. 确认后写入 `docs/engineering/04-tech-decisions.md`。
7. 再进入实现。

除非用户明确指定某个技术，并明确表示不需要比较方案，否则 AI 不能跳过本规则。

## 8.1 性能设计规则

AI 在设计方案和实现新功能时必须提前考虑性能：

1. 前端优先考虑按需引入、懒加载、代码分割和资源体积控制。
2. 大型组件、图表、编辑器、地图、富文本等默认不进入首屏主包，除非项目明确需要。
3. 列表、表格、日志、消息流等大数据场景必须考虑分页、虚拟滚动或增量加载。
4. 请求层必须避免重复请求、瀑布请求和过大的响应体。
5. 后端必须考虑查询索引、分页、缓存、批量处理、连接复用、超时和重试边界。
6. 新增依赖必须说明性能影响和是否支持按需引入。
7. 如果为了简单暂时不做性能优化，必须说明原因和后续补点。

## 9. 文档输出规则

当任务涉及文档或交付说明时，AI 应输出：

1. 修改了哪些文件。
2. 核心实现思路。
3. 运行或验证命令。
4. 已知限制或后续建议。

回答要直接、具体，不要堆砌空话。

## 9.1 页面手动测试步骤输出规则

完成一个需求、功能、变更或 bug 修复后，如果涉及用户可见页面、交互流程、接口数据在页面上的表现，AI 在最终回复中必须给出“页面手动测试步骤”。

测试步骤必须能让用户直接在浏览器页面中验证，不只是写命令。

输出应包含：

1. 进入哪个页面或地址。
2. 使用什么账号、角色或前置数据。
3. 点击哪些按钮或执行哪些操作。
4. 输入哪些测试数据。
5. 预期看到什么页面变化、提示、数据结果或错误状态。
6. 至少包含一个正常流程。
7. 如果功能有明显边界，补充 1-2 个异常/边界流程。

如果本次改动不涉及页面，例如纯后端、脚本、配置或文档，AI 必须说明“本次无页面手测步骤”，并给出可替代的验证方式。

## 10. 沟通规则

AI 与用户沟通时必须：

1. 先给结论，再给必要解释。
2. 不确定时明确说明不确定点。
3. 不假装已经验证未验证的内容。
4. 不把猜测写成事实。
5. 遇到多种方案时，给出推荐方案和取舍理由。
6. 用户要求直接实现时，不只停留在方案层。

## 11. 任务完成标准

一个开发任务只有在满足以下条件后才算完成：

1. 需求中的主要功能已经实现。
2. 相关错误和边界情况已处理。
3. 已运行必要验证，或明确说明无法验证的原因。
4. 没有引入明显无关改动。
5. 最终回复包含清晰的完成说明。

## 12. AI 必须主动避免的问题

以下行为默认禁止：

1. 大段重写用户没有要求修改的模块。
2. 用 mock 数据掩盖真实接口问题。
3. 删除测试来让结果通过。
4. 用 `any`、空 catch、硬编码超时等方式绕过类型或错误。
5. 将复杂业务逻辑写进 UI 组件而不拆分。
6. 把权限校验只放在前端。
7. 只看文件名猜业务含义。
8. 生成无法运行的伪代码后声称已完成。
